项目概览

甘草 Copilot 是一个本地优先的 AI 业务助手扩展。用户在业务网页旁打开 Side Panel，就能对话、诊断页面、上传资料、运行 OCR、触发 Browser Use 自动化、保存工作流并查看任务结果。

产品定位

围绕 Chrome 标签页完成信息获取、页面操作、资料问答和跨页面业务任务，而不是控制整台电脑。

技术栈

Manifest V3、React 18、TypeScript、Vite、Ant Design、IndexedDB、PaddleOCR.js、PDF.js、Vitest、Playwright。

核心能力

AI 对话、Browser Use、资料中心、OCR、页面诊断、页面监控、自定义命令、可复用工作流。

本地优先

聊天历史、资料、任务、Memory、OCR 结果和工作流默认保存在浏览器本地；API Key 只存入本机配置。

总体架构

架构按入口层、后台中枢、能力层、执行与数据层、外部服务五层组织。所有用户动作最终都会进入 background service worker，再由它决定调用模型、内容脚本、本地资料库、下载管理或监控调度。

运行入口

Vite 当前打包 6 个入口，对应 Manifest V3 扩展的页面、后台、内容脚本和 OCR 沙箱。public 目录会在构建结束时复制到 dist，并补齐 ONNX Runtime Web 的 wasm/mjs 资源。

入口 源码 职责 运行环境
sidePanel.js src/sidePanel/index.tsx 日常对话、附件解析、OCR、工具箱、任务卡、资料引用查看。 Chrome Side Panel extension page
dashboard.js src/dashboard/index.tsx 模型设置、工作流编辑、任务中心、自动化管理。 Extension page
background.js src/background/index.ts 消息路由、模型网关、任务执行器、Browser Use、下载、监控 alarm。 Manifest V3 service worker module
content.js src/content/index.ts 注入业务页面，收集页面语义、执行 DOM 动作、采集控制台错误。 Content script at document_start
ocrHost.js src/offscreen/ocrHost.ts 承接后台 OCR 任务，避免 Side Panel 关闭时任务中断。 Chrome offscreen document
paddleocrSandbox.js src/sidePanel/utils/paddleocrSandbox.ts 在 sandbox 页面中初始化 PaddleOCR 和 ONNX Runtime。 Extension sandbox page

Browser Use 闭环

Browser Use 是自动化能力的产品名称。内部仍保留 computer_use、ComputerUse* 和 RUN_COMPUTER_USE 作为兼容协议。它的关键原则是：LLM 负责语义目标和短期计划，TargetResolver 负责把目标落到真实页面元素。

目标解析从自然语言目标提取站点、动作、查询、风险级别和成功标准。
页面观察content script 返回元素、区域、页面状态、搜索结果和表格集合。
阶段计划按 ComputerUsePhase 推进，规则优先，LLM 兜底生成下一批动作。
目标定位优先使用集合、序号、父路径、阶段记忆，避免反复点击失败候选。
动作执行点击、输入、选择、滚动、下载、上传、结构化提取等原子动作。
验收校验用 URL、文本、元素、表格、下载结果或页面变化判定动作是否达成。
失败恢复保存 checkpoint、已完成阶段、下载结果、最后页面和阻塞原因。
用户确认保存、提交、删除、发送、支付等高风险动作必须等待用户允许。
结果交付输出摘要、结构化数据、下载资料、任务 trace 和可复用工作流草稿。
持续沉淀成功任务可保存为参数化 workflow；失败任务可从失败 phase 重试。

消息协议

Side Panel、Dashboard、Content Script 和 Background 通过 chrome.runtime.sendMessage / onMessage 通信。后台优先把任务类消息交给 handler，再按模型、页面工具、自动化和兼容协议分发。

任务消息

  • RUN_AUTOMATION_TASK 启动任务中心记录。
  • STOP_AUTOMATION_TASK 停止并标记任务。
  • RUN_PAGE_MONITOR_NOW 立即执行监控。
  • UPSERT_PAGE_MONITOR_ALARM 同步定时计划。

Browser Use 兼容消息

  • RUN_COMPUTER_USE 从聊天入口启动动态浏览器任务。
  • STOP_COMPUTER_USE 中止当前 runId。
  • CONFIRM_COMPUTER_USE_ACTION 响应高风险动作确认。
  • COMPUTER_USE_TRACE 记录观察、计划、动作和结果。

模型与工具消息

  • GET_MODEL_PROFILES / UPSERT_MODEL_PROFILE 管理模型。
  • TEST_MODEL_PROFILE 发起连接测试。
  • EXECUTE_TOOL 执行业务工具。
  • EXECUTE_BROWSER_TOOL 调用当前页面工具。
协议设计要点 消息返回统一倾向于 { success, ... } 或应用错误 payload；模型、OCR、认证和网络错误会通过 appErrors.ts 映射成用户可理解的恢复建议。

数据与存储

插件采用本地优先策略：结构化资料进入 IndexedDB，配置与任务记录进入 chrome.storage.local。需要发送给模型的内容由用户动作触发，并由模型配置决定目标服务。

IndexedDB store 内容
assets资料元数据、标题、来源、解析状态、OCR 状态。
assetContents资料正文、表格、OCR 结构化结果、页码/章节等引用信息。
chunks按资料拆分的检索分块，带 assetId 索引。
results需求拆解、网页结构化数据、资料问答等任务产物。
rawFiles本地原始文件 Blob，用于重新解析和打开来源。

chrome.storage.local

模型配置、活动模型 ID、工作流、草稿、自定义命令、任务运行记录、页面监控定义、Memory 候选和登录态兼容数据。

数据升级原则

不清空已有资料、Memory、任务记录和工作流；旧 documentDb/documentStore 保留兼容 re-export，统一落到 DocumentRepository。

模型网关

ModelGateway 是 background-only 的模型访问层。Side Panel 不直接持有 API Key，也不直接请求模型服务；所有流式对话、JSON 规划、文本补全、文件上传和连接测试都从后台发起。

模型配置

支持 DeepSeek、甘草业务模型和 OpenAI-compatible。配置项包含 Base URL、model、API Key 与 streaming/tools/json/files 能力开关。

能力约束

Browser Use 的计划和意图需要 JSON 能力；资料原生上传需要 files 能力。能力不足会返回 MODEL_CAPABILITY_MISSING。

脱敏与取消

错误信息会用 redactSecrets 去除 API Key 和 Authorization；切换模型或停止生成会取消当前请求。

ModelGateway
  -> getProfile(profileId)
  -> GLMClient streaming chat
  -> /chat/completions JSON planning
  -> /chat/completions text completion
  -> /files native upload when capability files = true

资料中心与 OCR

资料中心把上传文件、网页结构化数据、下载文件和 OCR 结果统一成可检索资产。OCR 使用 PaddleOCR.js + ONNX Runtime Web，在 sandbox 中运行，后台可通过 offscreen document 继续长任务。

文件解析链路

  • 支持 PDF、Word、Excel、CSV、Markdown、JSON、文本、图片等常见格式。
  • PDF.js、mammoth、xlsx 等本地解析工具先提取文本和表格。
  • 扫描件或图片型资料标记为 ocrStatus: pending。
  • 解析内容进入 DocumentRepository 后重建 chunks。

OCR 链路

  • 模型文件来自 public/paddleocr/models。
  • ONNX Runtime wasm/mjs 会复制到 dist/paddleocr/ort。
  • OCR 输出包含原文、字段、表格、章节、页码和低置信度提示。
  • 人工校正结构化 OCR 后会保留原文并更新检索索引。

工作流与监控

任务中心统一承载 7 类任务：computer_use、workflow、page_monitor、page_diagnosis、document_qa、ocr、extract。固定工作流可以调用普通页面工具，也可以把成功的 Browser Use 任务沉淀为 computerTask 步骤。

AutomationRunner

顺序执行 observe、navigate、click、type、forms、assert、extract、screenshot、computerTask 等固定步骤，并向 UI 推送进度事件。

TaskExecutorRegistry

按任务 kind 注册执行器，统一 validate、run、stop、progress 和 TaskResult。Service Worker 重启后遗留 running 任务会安全收口。

Page Monitor

支持内容变化、包含内容、数值阈值、新增记录和状态转换；命中后可投递 Chrome 通知、飞书、钉钉或 Webhook。

安全边界

扩展权限覆盖侧边栏、标签页、存储、脚本注入、下载、alarm、offscreen 和通知。可选 debugger 只用于增强页面诊断，默认路径不依赖它。

Manifest 权限

sidePanel、tabs、storage、scripting、activeTab、downloads、alarms、offscreen、notifications。

optional: debugger host: <all_urls>

风险控制

  • 高风险动作必须由用户确认后执行。
  • LLM 不直接猜 selector，动作前后必须重新观察和校验。
  • API Key 脱敏显示，不写入 trace、导出结果或源码。
  • Chrome 内置页、Web Store、强验证码和受限 iframe 明确阻塞。
CSP 与沙箱 扩展页允许 wasm-unsafe-eval 以支持本地 OCR 运行时；PaddleOCR 的 eval 需求被隔离在 paddleocrSandbox.html 的 sandbox 页面中。

开发运行

本地开发以 pnpm、Vite 和 Chrome 扩展加载流程为主。构建产物位于 dist/，需要从 Chrome 扩展管理页加载已解压目录。

# 安装依赖
pnpm install

# 生产构建
pnpm build

# 加载 Chrome 扩展
# chrome://extensions/
# 开启开发者模式
# 选择 dist/ 目录

# 持续监听构建
pnpm watch
  • Node.js 18 或更高版本。
  • Chrome 或 Chromium 浏览器。
  • 模型设置中配置 Base URL、模型名称和 API Key。
  • 普通 http/https 页面才能完整测试内容脚本和自动化。
  • 修改 OCR 资源后确认 det/rec 模型与 ORT wasm 已被复制。

测试门禁与排障

项目已有 TypeScript、Vitest、Vite 构建和 Playwright 扩展 E2E。新增能力建议覆盖共享纯函数、后台执行器、内容脚本工具和真实扩展路径。

常用脚本

pnpm test
pnpm build
pnpm test:e2e
pnpm verify

排障入口

  • 模型不可用：检查模型配置、能力开关、Base URL、API Key 和配额。
  • 页面不可操作：确认当前页面允许内容脚本注入，并查看 pageState 与 pageSignals。
  • OCR 失败：检查 sandbox、模型文件和 dist/paddleocr/ort 资源。
  • 下载未入库：查看 downloads 权限、临时 URL 限制和 DownloadManager 结果。
